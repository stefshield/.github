# Governance

This document describes how StefShield projects are maintained and how decisions are made. It reflects the organization's current state and is designed to evolve as the project grows.

## Current Structure

StefShield is currently maintained by a single maintainer who holds final decision-making authority across all repositories. This includes code, direction, releases, and community standards.

This model keeps things simple and accountable at this stage of the project.

## Decision Types

Decisions are grouped into three tiers based on their scope and impact.

**Routine decisions**
Handled by the maintainer as part of day-to-day project maintenance:

- Merging pull requests
- Closing or triaging issues
- Bug fixes and minor improvements
- Documentation updates

**Significant decisions**
A discussion period of 24–72 hours is encouraged for significant decisions unless urgency requires faster action:

- Breaking changes or API redesigns
- Major new features
- Dependencies with significant scope or security impact
- Changes to project conventions or tooling

**Structural decisions**
Reserved exclusively for the top-level maintainer(s) due to their long-term impact on the project:

- Adding or removing maintainers
- Changing the project license
- Archiving or deprecating a repository
- Changes to this governance document
- Security policy or response process changes (if applicable)

## Becoming a Maintainer

As StefShield grows, trusted contributors may be invited to take on maintainer responsibilities.

Maintainership is not applied for — it is earned through sustained, demonstrated contribution. Relevant factors include:

- Consistent, high-quality contributions over time
- Constructive participation in reviews and discussions
- Good judgment and alignment with project goals and values
- Reliability and follow-through

Invitations to maintainership are made at the discretion of the existing maintainer(s).

There is no fixed checklist or timeline; evaluation is holistic.

## Conflict Resolution

Disagreements are discussed openly in issues or pull requests. If consensus is not reached, the maintainer will make a final decision in the best interest of the project and its users.

## Changes to This Document

Changes to this governance document are treated as structural decisions and require maintainer approval. Proposed changes should be submitted as a pull request and will be reviewed before merging.
