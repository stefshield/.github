# Security Policy

StefShield takes security seriously and appreciates responsible disclosure.
If you discover a vulnerability in any of our projects, please follow the process below to report it responsibly.

## Scope

This policy applies to vulnerabilities found in code maintained in StefShield repositories.

The following are out of scope:

- Vulnerabilities in third-party dependencies — please report those upstream
- Issues that require unrealistic or highly privileged attacker models without a clear security impact
- Non-security bugs — use the standard issue tracker for those

## Supported Versions

Security updates are generally applied to the latest maintained versions of our projects.
Older or unsupported versions may not receive fixes.

## Reporting a Vulnerability

**Do not open a public GitHub issue for security vulnerabilities.**

Report vulnerabilities privately by emailing:

**[stefshield.team@gmail.com](mailto:stefshield.team@gmail.com)**

Alternatively, you may use GitHub's private vulnerability reporting feature (Security Advisories) if enabled for the repository.

Please include as much of the following as possible:

- A description of the vulnerability and its potential impact
- The affected repository and version (if known)
- Steps to reproduce or a proof of concept
- Any suggested mitigations, if applicable

## Safe Harbor

We support and encourage security research conducted in good faith.
If you follow this policy and act responsibly, we will not take legal action against you for your research.
Please avoid actions that could harm users, disrupt services, or compromise data.

## Response Process

Once a report is received:

1. We aim to acknowledge receipt within **72 hours**
2. We will investigate and provide a status update within **7 days**
3. We will work with you to understand and resolve the issue
4. Fix timelines depend on complexity and severity — we will keep you informed

We appreciate your patience. Security issues are treated as a priority.

## Coordinated Disclosure

We follow a coordinated disclosure process.

Please do not publicly disclose the vulnerability until we have had a chance to
investigate and release a fix. We will work with you on disclosure timing and aim
to resolve issues as quickly as possible.

If we are unresponsive for more than **14 days**, you may proceed with disclosure
at your discretion, ideally minimizing potential risk to users.

## Attribution

We are happy to credit reporters in the relevant release notes or security advisory, unless you prefer to remain anonymous.
